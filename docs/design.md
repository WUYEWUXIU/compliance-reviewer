# 关键设计说明

## 一、技术路径选择

本系统采用 **RAG + Prompt Engineering** 双路增强。

| 路径               | 是否采用 | 理由                                                                                                   |
| ------------------ | -------- | ------------------------------------------------------------------------------------------------------ |
| RAG                | ✅       | 监管条文数量有限但高度专业，LLM 训练语料无法覆盖最新法规；RAG 确保每条判断有可追溯的条文依据，消除幻觉 |
| Prompt Engineering | ✅       | 通过强约束 JSON Schema + 违规类型枚举（V01-V09），将 LLM 输出收束为结构化结论，便于解析和展示          |
| Agent Workflow     | ❌       | 本任务是单轮确定性判断，无需多步工具调用；引入 Agent 链会增加延迟和不稳定性，收益低于成本              |

---

## 二、知识库构建：按条文粒度分块 + 上下文依赖处理

### 2.1 分块策略

监管文档按**条文**（article）粒度分块，每个 chunk 对应一条具体规定。

**为何不按段落或固定字数分块？**

- 监管条文是最小完整语义单元，段落切分会破坏条文内部的逻辑关系（"不得……除非……"这类条件句）
- 按条文分块后，检索命中的 chunk 可直接映射到"第 X 条"，输出结果可作为法律依据引用
- 固定字数分块在长条文处会截断关键语句，短条文则引入噪声上下文

每个 chunk 携带 `article_id`、`chapter`、`violation_tags` 字段，检索结果可直接附带元数据用于 Prompt 构建。

### 2.2 条款上下文依赖问题

监管文档存在大量**条文间依赖**：某一条款的适用性可能受其他条款的例外、定义或前置条件约束。例如：

- **例外依赖**：条文 A 规定"不得承诺保本保息"，条文 B 补充"但结构性存款除外"——只检索到 A 会导致误判。
- **定义依赖**：条文 C 禁止"虚假宣传"，但"虚假宣传"的界定散落在其他条文中，单独引用 C 可能判断失准。
- **层级依赖**：通用禁止条款与特定产品（如互联网保险）的专项规定之间存在适用优先级。

**处理策略**：

| 策略                                  | 实现方式                                                                             | 作用                                |
| ------------------------------------- | ------------------------------------------------------------------------------------ | ----------------------------------- |
| **关联条文索引**                | 分块时为每个 chunk 附加 `related_articles` 字段，定义来源、上位/下位法关系         | 检索命中时自动将关联条文加入上下文  |
| **检索扩展（Query Expansion）** | 检索到 Top-K 条文后，遍历其 `related_articles`，将未命中但高关联的条文补充进候选集 | 弥补单路检索的遗漏                  |
| **Prompt 内标注逻辑关系**       | 注入 LLM 的条文按原文顺序排列，并在关联处添加批注（如"【例外】参见第 X 条"）         | 帮助 LLM 理解条款间的条件与例外关系 |

上下文依赖的处理目标是：**宁可多给 LLM 几条相关条文，也不能让 LLM 基于孤立的半条法规做判断。**

## 三、混合检索：BM25 + 向量 + RRF + Rerank

```
BM25 稀疏检索 Top-30
                      ──→ RRF 融合 → Top-60 → gte-rerank 精排 → Top-10 → LLM
FAISS 向量检索 Top-30
```

- **BM25**：精确命中监管术语（"保本保息"、"犹豫期"等关键词），对低频专业词汇优势明显
- **向量检索**（text-embedding-v3）：覆盖语义相近的隐式违规表述，如"历史业绩保障"→"承诺保本"
- **RRF 融合**（k=60）：无需训练，稳定合并两路排序，避免单路漏检
- **Rerank**（gte-rerank）：cross-encoder 对 Top-60 再精排，取 Top-10 送入 LLM，显著提升相关性

## 四、Query 改写：捕捉隐式违规

审核文本在进入检索前，先经过 **LLM（qwen-turbo）语义改写**，生成若干面向监管条文的检索意图查询。

**核心价值**：隐式违规无法靠关键词匹配发现。

| 原始文案                   | 直接检索                      | 改写后查询                             |
| -------------------------- | ----------------------------- | -------------------------------------- |
| 多年来客户本金从未出现亏损 | 无"保本"关键词，BM25 无法命中 | "承诺本金不受损失"→ 命中 V01 相关条文 |
| 业内首款通过银保监核准产品 | 无明显违规词                  | "伪造监管批准信息"→ 命中 V08 相关条文 |

改写结果中同时标注违规类型 ID（V01-V09），作为精排阶段的类型先验信息。

## 五、Prompt 结构与输出约束

```
[System]
你是金融合规审核专家。以下是与审核相关的监管条文：
{检索到的 Top-10 条文原文，含条文编号和出处}

审核规则：
1. 严格基于上述条文判断，不得引用条文以外的依据
2. 输出严格的 JSON，字段：compliant / violations[] / suggestions / confidence
3. 每条 violation 必须包含 violation_type_id（V01-V09）、cited_article、reason

[User]
请审核以下营销文案：
{营销文案原文}
```

**关键约束**：

- 违规类型 ID 限定在 V01-V09 枚举内，防止 LLM 自造类型
- System Prompt 中注入条文原文（非摘要），保证引用可回溯

## 六、多模态支持

图文输入时，先用 `qwen-vl-max` 对图片做 OCR 与内容理解，提取文字描述后与文本合并，进入统一审核流程。文本审核与图像理解解耦，互不影响。

---

## 七、效果评估模块

标注了两批 Golden Set 样本（`tests/golden_set/`）：

- `short.json`：覆盖 V01-V09 每类违规的典型正例 + 否定式反例 + 边界难判 case
- `long.json`：多违规叠加的复合 case、隐式违规等高难度样本

核心评估指标：

| 指标                   | 含义                            |
| ---------------------- | ------------------------------- |
| `compliant_accuracy` | 合规/违规二分类准确率（主指标） |

运行评估并自动生成 Markdown 报告：

```bash
python scripts/evaluate.py --golden-set tests/golden_set/short.json
# 报告保存至 tests/evaluation_report/report_<timestamp>.md
```
