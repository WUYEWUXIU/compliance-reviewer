# RAG 系统执行大纲

------

## 0. 前置准备：违规类型枚举表

**0.1 人工研读三份监管文档**

- 通读《保险销售行为管理办法》《金融产品网络营销管理办法》《互联网保险业务监管办法》
- 摘录所有涉及禁止行为、强制要求、用语规范的条文

**0.2 构建封闭违规类型枚举表**

- 归纳 10-20 个违规类型，每类包含：
  - 编号（V01-V20）
  - 类型名称
  - 典型关键词/表现形式
  - 对应主要条文范围（初步映射，后续由标签精化）
- 示例：

| 编号 | 违规类型       | 典型表现                 |
| ---- | -------------- | ------------------------ |
| V01  | 承诺保本保息   | "保本"、"稳赚"、"零风险" |
| V02  | 夸大预期收益   | "年化X%稳稳到手"         |
| V03  | 使用绝对化用语 | "最优"、"第一"、"最强"   |
| V04  | 无资质代言     | 明星/KOL推荐无资质披露   |
| V05  | 缺失风险提示   | 无法定风险提示语         |
| V06  | 误导性比较     | 与其他产品做不实对比     |
| ...  | ...            | ...                      |

**0.3 枚举表的下游用途**

- Chunking 阶段：给条文打标签的类目来源
- Query 改写阶段：改写输出约束为从枚举表中选择
- LLM 审核阶段：system prompt 附入枚举表，强约束输出
- 评估阶段：违规类型准确率的 ground truth 参照

------

## 1. 文档预处理与 Chunking

**1.1 文档解析**

- 用 PyMuPDF 提取三份 PDF 原始文本
- 识别文档层级结构：章标题 → 条编号 → 条文正文
- 处理解析噪音：页眉页脚、页码、换行断字

**1.2 条文粒度切割**

- 以「第X条」为分割边界，每条作为一个独立 chunk
- 每个 chunk 携带完整 metadata：
  - `doc_name`：所属文档名
  - `chapter`：所属章节标题
  - `article_id`：条文编号
  - `article_text`：条文完整正文
- 统计三份文档总条数，检查有无漏切或错切

**1.3 显式引用关系解析（为 Graph 构建准备）**

- 用正则扫描所有含「前款」「本条」「依照第X条」「第X条规定」的条文
- 构建有向引用关系表：`(source_article_id, target_article_id)`
- 标记被多条引用的「核心条文」，后续图扩展优先级更高

**1.4 违规类型标签标注**

- 对每个 chunk，从 §0.2 枚举表中选择 1-3 个匹配的违规类型编号写入 metadata
- 标注方式：优先人工标注；条文数量大时用 LLM 辅助，但输出严格约束为枚举表编号，人工抽检验证
- 无法匹配任何类型的条文标记为 `V00`（通用合规要求），不丢弃

------

## 2. 索引构建

**2.1 轻量知识图谱（处理显式引用依赖）**

- 使用 networkx 构建有向图，节点为 `article_id`，边为 §1.3 解析的引用关系
- 图持久化为 JSON（节点列表 + 边列表）
- 仅处理显式引用（类型A），语义隐含依赖不在 demo 阶段处理

**2.2 Dense 向量索引**

- 使用百炼 `text-embedding-v3` 对 `article_text` 做 embedding
- 向量库使用 FAISS，索引时同步存入 chunk_id 与完整 metadata
- 索引文件序列化到本地

**2.3 Sparse BM25 索引**

- 使用 `rank_bm25` 库构建索引
- 中文分词使用 jieba，添加自定义词典保留监管专业术语（"误导性陈述"、"保本保息"、"投保人"等不切分）
- BM25 索引序列化，与向量索引共享 chunk_id 体系

------

## 3. Query 理解：违规意图改写

**3.1 改写 Prompt 设计**

- 输入：原始营销文案
- 任务：识别文案中潜在违规行为
- 输出约束：从 §0.2 枚举表中选择匹配的违规类型编号，并为每个类型生成 1 个检索短语（10 字以内，贴近监管用语）
- few-shot 示例覆盖常见违规类型

**3.2 改写结果校验**

- 输出的违规类型编号必须在枚举表范围内，否则过滤
- 有效改写结果 < 2 个时，降级为直接用原文发起检索

**3.3 多 Query 并行检索**

- 每个改写短语独立发起检索（§4.1）
- 记录每个 query 的检索结果及对应违规类型编号，用于后续 `multi_query_agreement` 计算

------

## 4. 检索、图扩展与精排

**4.1 双路检索执行**

- 每个 query：BM25 取 Top-10，向量检索取 Top-10
- 两路结果合并去重，进入 RRF 融合

**4.2 RRF 融合**

- 公式：`score(d) = Σ 1/(k + rank(d))`，k=60
- 同一 chunk 在多个 query 中出现则累加得分
- 输出跨 query 的融合排序列表

**4.3 图扩展（一跳）**

- 对 RRF Top-10 结果中的每个 chunk，在知识图谱中查询其一跳邻居
- 将邻居节点对应的 chunk 加入候选池，标记来源为「图扩展」
- 图扩展 chunk 的 RRF 分数设为触发它的父节点分数 × 衰减系数（如 0.7）

**4.4 Rerank 精排**

- 使用百炼 `gte-rerank`，输入为 (原始营销文案, chunk_text) 对
- 候选池（RRF Top-10 + 图扩展结果）送入 rerank，输出精排分数
- 取 Top-5 作为最终送入 LLM 的 context
- **保存每个 chunk 的 rerank 分数**，作为置信度计算输入

**4.5 检索结果过滤**

- 设置 rerank 分数最低阈值（如 0.3）
- 全部低于阈值时：不送入 LLM，直接返回「未检索到相关条文」，并标注置信度极低
- 部分低于阈值时：仅送入达标的 chunk

------

## 5. 置信度计算

> LLM 不参与置信度打分，全部由系统内部可观测量计算。

**5.1 四个分量定义**

| 分量                    | 计算方式                                           |
| ----------------------- | -------------------------------------------------- |
| `rerank_score`          | Top-1 chunk 的 rerank 分数，归一化到 0-1           |
| `chunk_coverage`        | Top-5 中指向同一违规类型的 chunk 数量 / 5          |
| `multi_query_agreement` | 多个改写 query 均检索到同一条文的比例              |
| `violation_type_match`  | LLM 输出的违规类型是否在枚举表中精确命中（0 或 1） |

**5.2 加权合成**

```
confidence = w1 * rerank_score
           + w2 * chunk_coverage
           + w3 * multi_query_agreement
           + w4 * violation_type_match
```

- 初始权重：w1=0.4, w2=0.2, w3=0.2, w4=0.2
- 权重在评估阶段用 golden set 校准

**5.3 合规结论为「合规」时的特殊处理**

- 若 rerank_score 全部低于阈值导致无 context，合规结论置信度强制压低（≤ 0.4），并在输出中标注「因未检索到相关条文而判定合规，结论可信度低」
- 避免「没找到条文 = 一定合规」的误判

------

## 6. LLM 审核

**6.1 System Prompt 设计**

- 角色定义：金融合规审核专家
- 附入 §0.2 违规类型枚举表（完整）
- 明确任务：仅根据提供的条文 context 判断，禁止引用 context 之外的条文
- 输出格式强约束（JSON schema）：

```
{
  "compliant": "yes/no",
  "violations": [
    {
      "violation_type_id": "V02",
      "violation_type_name": "夸大预期收益",
      "article_id": "第二十一条",
      "doc_name": "保险销售行为管理办法",
      "article_text": "条文原文",
      "reason": "具体违规说明"
    }
  ]
}
```

**6.2 User Prompt 组装**

- 结构：营销文案 + 分隔符 + Top-5 条文（按 rerank 分数从高到低，每条附编号、来源文档、正文）

**6.3 输出解析与兜底**

- JSON 解析失败触发重试（最多 2 次，prompt 中强化 JSON 格式要求）
- 最终仍失败则返回结构化错误信息

**6.4 置信度附加**

- 将 §5.2 计算的 confidence 值附加到最终输出 JSON 中
- LLM 输出结果与置信度合并为最终响应

------

## 7. 评估模块

**7.1 Golden Set 构建**

- 人工构造 30 条测试样本：
  - 明确违规（15条）：覆盖枚举表中至少 8 种违规类型，每类 ≥ 1 例
  - 明确合规（10条）：措辞规范、含法定风险提示
  - 边界模糊（5条）：考察置信度是否合理降低
- 每条标注：`expected_compliant`、`expected_article_ids`、`expected_violation_type_ids`

**7.2 评估指标**

- **合规判断准确率**：yes/no 与标注一致的比例
- **条文命中率**：引用条文 ID 与标注的交集 / 标注总数
- **违规类型准确率**：输出的 violation_type_id 是否与标注一致
- **置信度校准**：高置信度（>0.8）样本准确率是否显著高于低置信度（<0.5）样本

**7.3 评估流程**

- 批量跑完 golden set，结果存为 JSON
- 自动计算四项指标并输出报告
- 输出错误案例列表，便于定位是 chunking / 检索 / prompt 哪个环节的问题

**7.4 迭代闭环**

- 条文命中率低 → 优化 chunking 粒度或 Query 改写 prompt
- 违规类型准确率低 → 检查枚举表定义是否清晰，优化 LLM system prompt
- 置信度失真 → 用 golden set 重新校准 w1-w4 权重
- 合规判断准确率低 → 增加 few-shot 或调整 rerank 阈值

------

## 执行优先级

```
P0 核心链路（必须跑通）
  §0 枚举表 → §1.1-1.2 Chunking → §2.2-2.3 双路索引
  → §3 Query改写 → §4.1-4.2 RRF → §4.4 Rerank
  → §5 置信度计算 → §6 LLM审核

P1 质量提升（显著提升可信度）
  §1.3-1.4 引用解析+标签 → §2.1 知识图谱 → §4.3 图扩展
  → §7.1-7.3 评估模块

P2 迭代优化（演示加分项）
  §5.3 合规置信度特殊处理 → §4.5 检索过滤 → §7.4 迭代闭环
```
