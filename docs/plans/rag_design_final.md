# 合规审核 RAG 系统 — 最终设计

> 本文档为 `rag_design.md` 和 `rag_design_highlevel.md` 的综合定稿版。
> 经业务合规、检索架构、工程实现三个视角交叉审阅，修正了原文中的 CRITICAL 缺陷。

---

## 1. 系统定位

### 1.1 背景与目标

金融营销文案须符合《保险销售行为管理办法》《金融产品网络营销管理办法》《互联网保险业务监管办法》等监管要求。人工逐条审核效率低、标准不一致。

**本系统目标：** 给定一段营销文案，自动定位潜在违规内容、引用对应监管条文、给出合规结论，并附上系统自评的置信度分数。

### 1.2 系统边界

```
输入：营销文案（文本）
         │
         ▼
  ┌─────────────┐
  │ 合规审核RAG  │
  └─────────────┘
         │
         ▼
输出：合规/违规结论 + 引用条文 + 违规类型 + 置信度
```

**系统不做：** 自动修改文案、对接审批流程、实时监控流量。

## 2. 核心架构

### 2.1 模块总览

系统由五个主要模块组成，形成完整的审核闭环：

```
┌─────────────────────────────────────────────────────────┐
│                      离线阶段                            │
│                                                         │
│  监管文档  ──▶  [文档解析与切割]  ──▶  [知识索引构建]    │
│  违规枚举表 ─────────────────────────────────────────▶  │
└─────────────────────────────────────────────────────────┘
                                          │
                                          ▼
┌─────────────────────────────────────────────────────────┐
│                      在线阶段                            │
│                                                         │
│  营销文案 ──▶ [Query理解] ──▶ [混合检索+精排] ──▶ [LLM审核] ──▶ 结论
│                                      │                  │
│                                      └──▶ [置信度计算] ──┘
└─────────────────────────────────────────────────────────┘
```

| 模块 | 职责 | 产出 |
|------|------|------|
| 文档解析与切割 | 将监管文档解析为条文级别的知识单元 | 结构化条文库 |
| 知识索引构建 | 建立向量索引、关键词索引、条文引用图谱 | 可检索知识库 |
| Query 理解 | 将营销文案转化为面向监管的检索意图 | 结构化检索请求 |
| 混合检索+精排 | 召回相关条文并按相关性排序 | Top-N 条文候选 |
| LLM 审核 | 基于检索到的条文给出合规判断 | 结构化审核结论 |
| 置信度计算 | 从系统内部可观测信号合成可信度分数 | 置信度分值 |

## 3. 关键设计决策

### 3.1 封闭违规类型枚举表

**决策：** 人工预定义违规类型（V01–V20），系统全链路共享。枚举表覆盖《保险销售行为管理办法》《金融产品网络营销管理办法》《互联网保险业务监管办法》中的高频红线。

**为什么：**
- 让检索、审核、评估三个阶段使用统一语言，避免语义漂移
- 约束 LLM 输出范围，减少幻觉
- 为评估提供精确的 ground truth 对照

**权衡：** 枚举表需人工维护，监管政策更新时需同步修订。新型违规可能落入 V00（未分类）。

#### 3.1.1 枚举表初版（11类）

| 编号 | 违规类型 | 典型表现/话术 | 主要对应条文 |
|------|----------|---------------|-------------|
| V01 | 承诺本金不受损失 | "保本、保息、零风险、本金无忧" | 保险销售第21条、金融网销第8条 |
| V02 | 承诺确定收益 | "年化X%稳稳到手、固定收益、确定回报" | 保险销售第21条、金融网销第8条 |
| V03 | 使用绝对化用语 | "最优、第一、最强、绝无仅有" | 金融网销第11条、广告法第9条 |
| V04 | 无资质或不当代言 | 明星/KOL推荐无资质披露；员工冒充专家 | 保险销售第28条、金融网销第12条 |
| V05 | 缺失风险提示 | 无"保险产品由保险公司承保"等法定提示 | 保险销售第23条、金融网销第9条 |
| V06 | 误导性产品比较 | 与其他产品做不实对比、贬低同业 | 保险销售第26条、金融网销第11条 |
| V07 | 隐瞒/淡化费用 | "首期免保费、零手续费、无管理费" | 保险销售第21条、第25条 |
| V08 | 诱导退保/转保 | "旧保单不划算，退了买新的、升级换代" | 保险销售第27条 |
| V09 | 伪造/篡改备案信息 | "已获银保监会特别批准、备案编号XXX" | 互联网保险第34条 |
| V10 | 不当使用客户信息 | 未经同意使用客户案例、暗示内部客户专享 | 金融网销第9条、个人信息保护法 |
| V11 | 违规承诺增值服务 | "买就送体检、保单随时变现、免费旅游" | 保险销售第22条、各地监管通报 |
| V00 | 未分类/通用合规要求 | 无法归入上述类型的条文 | 兜底 |

> **注：** V01 和 V02 在原文中合并为"承诺保本保息"，审阅发现该合并导致边界模糊。拆分后 V01 侧重本金安全承诺，V02 侧重收益确定性承诺。

---

### 3.2 混合检索（语义 + 关键词）

**决策：** 同时使用向量检索（语义相似）和 BM25 检索（关键词匹配），结果通过 RRF 融合。

**为什么：**
- 监管文本含大量专业术语，关键词检索捕捉精确措辞
- 违规行为表述多样，向量检索覆盖语义近义表达
- 单路检索在合规场景中召回率不足

**权衡：** 维护两套索引，系统复杂度略高。Demo 阶段保留双路，若资源受限可降级为单路。

---

### 3.3 条文引用知识图谱（条件性使用）

**决策：** 解析条文间的显式引用关系（"依照第X条""第X条规定"），构建有向图。但图扩展不无条件触发，仅在满足条件时作为「参考条文」补充到 LLM prompt 中。

**为什么：**
- 监管条文存在大量交叉引用，仅检索表面命中条文会丢失关联义务
- 但监管引用关系是"义务指向"而非"语义相似"，无条件扩展会引入噪音

**权衡：** 仅处理显式引用，隐含语义关联暂不覆盖。

---

### 3.4 系统自计算置信度（不依赖 LLM 自评）

**决策：** 置信度由系统内部四个可观测信号加权合成，LLM 不参与打分。

**为什么：**
- LLM 自我评估置信度不可靠（容易过度自信）
- 系统信号（检索分数、多 Query 一致性、条文覆盖度）客观且可解释
- 为业务干系人提供可信的"可信度仪表盘"

**权衡：** 初始权重为启发值，正式运行前需用标注数据校准。

```

```

## 5. 模块详细设计

### 5.0 违规类型枚举表（前置准备）

#### 5.0.1 人工研读三份监管文档

- 通读《保险销售行为管理办法》《金融产品网络营销管理办法》《互联网保险业务监管办法》
- 摘录所有涉及禁止行为、强制要求、用语规范的条文

#### 5.0.2 构建封闭违规类型枚举表

- 归纳 11 个违规类型（V01-V11）+ 1 个兜底（V00），详见 §3.1.1
- 每类包含：编号、类型名称、典型关键词/话术、对应主要条文范围

#### 5.0.3 枚举表的下游用途

- Chunking 阶段：给条文打标签的类目来源
- Query 改写阶段：改写输出约束为从枚举表中选择
- LLM 审核阶段：system prompt 附入枚举表，强约束输出
- 评估阶段：违规类型准确率的 ground truth 参照

---

### 5.1 文档预处理与 Chunking

#### 5.1.1 文档解析

- 用 PyMuPDF 提取三份 PDF 原始文本
- 识别文档层级结构：章标题 → 条编号 → 款编号 → 项
- 处理解析噪音：页眉页脚、页码、换行断字

#### 5.1.2 条文粒度切割（修正：条+款两级）

**策略：** 以「条」为一级边界、「款」为二级边界：
- 若条文内部有明确分款（第一款、第二款…），则以「款」为最小 chunk 单元
- 若条文无分款，则整条作为一个 chunk
- 每个 chunk 携带完整 metadata：
  - `doc_name`：所属文档名
  - `chapter`：所属章节标题
  - `article_id`：条文编号
  - `paragraph_id`：款编号（可选，无分款时为空）
  - `article_text`：该 chunk 的完整正文

> **修正原因：** 监管条文的「款」是独立的法律义务单元，不同款对应不同违规场景。以整条为单位会导致无关信息淹没真正相关的款，降低检索精度和 LLM 判断质量。

- 统计三份文档总条数/款数，检查有无漏切或错切

#### 5.1.3 显式引用关系解析（为 Graph 构建准备）

- 用正则扫描所有含「前款」「本条」「依照第X条」「第X条规定」「第X条第Y款」的条文
- 构建有向引用关系表：`(source_chunk_id, target_chunk_id)`
- 标记被多条引用的「核心条文」，后续图扩展优先级更高

#### 5.1.4 违规类型标签标注

- 对每个 chunk，从枚举表中选择 0-3 个匹配的违规类型编号写入 metadata
- 标注方式：优先人工标注；条文数量大时用 LLM 辅助，但输出严格约束为枚举表编号，人工抽检验证
- 无法匹配任何类型的条文标记为 `V00`（通用合规要求），不丢弃

---

### 5.2 索引构建

#### 5.2.1 轻量知识图谱（条件性使用）

- 使用 networkx 构建有向图，节点为 `chunk_id`，边为 §5.1.3 解析的引用关系
- 图持久化为 JSON（节点列表 + 边列表）
- 仅处理显式引用，语义隐含依赖不在 demo 阶段处理

#### 5.2.2 Dense 向量索引

- 使用百炼 `text-embedding-v3` 对 `article_text` 做 embedding
- 向量库使用 FAISS，索引时同步存入 chunk_id 与完整 metadata
- 索引文件序列化到本地

#### 5.2.3 Sparse BM25 索引

- 使用 `rank_bm25` 库构建索引
- 中文分词使用 jieba，添加自定义词典保留监管专业术语（"误导性陈述"、"保本保息"、"投保人"等不切分）
- **新增：** 配置中文停用词表（含「应当」「不得」「保险」「产品」「金融」等高频无区分度词），避免干扰 BM25 打分
- BM25 索引序列化，与向量索引共享 chunk_id 体系

---

### 5.3 Query 理解：违规意图改写

#### 5.3.1 改写 Prompt 设计

- 输入：原始营销文案
- 任务：识别文案中潜在违规行为
- 输出约束：从枚举表中选择匹配的违规类型编号，并为每个类型生成 1 个检索短语（10字以内，贴近监管用语）
- few-shot 示例覆盖常见违规类型

#### 5.3.2 改写结果校验

- 输出的违规类型编号必须在枚举表范围内，否则过滤
- 有效改写结果 < 2 个时，降级为提取原文中的「监管敏感词」（收益承诺词、绝对化用语、产品类型等）作为关键词 query，而非整段原文

> **修正原因：** 整段原文包含大量口语化噪音，与监管条文正式用语差距大，直接检索召回率显著低于改写后检索。提取敏感词作为 fallback 保留了检索意图。

#### 5.3.3 多 Query 并行检索

- 每个改写短语独立发起检索（§5.4.1）
- 记录每个 query 的检索结果及对应违规类型编号，用于后续 `multi_query_agreement` 计算

---

### 5.4 检索、图扩展与精排

#### 5.4.1 双路检索执行

- 每个 query：BM25 取 Top-10，向量检索取 Top-10
- 两路结果合并去重，进入 RRF 融合

#### 5.4.2 RRF 融合

- 公式：`score(d) = Σ 1/(k + rank(d))`
- **修正：** k 从 60 调整为 20。小语料库（三份文档共约 100-200 条/款）中，k=60 会过度平滑排名差异，降低 BM25 和向量检索的区分度
- 同一 chunk 在多个 query 中出现则累加得分
- 输出跨 query 的融合排序列表

#### 5.4.3 图扩展（条件性，修正）

- 对 RRF Top-10 结果中 rerank 分数 > 0.5 的 chunk，在知识图谱中查询其一跳邻居
- **修正：** 图扩展结果不直接加入精排候选池，而是作为「参考条文」在 LLM prompt 中单独标注（不占用 Top-5 名额）
- 图扩展 chunk 标注来源为「引用上下文」

> **修正原因：** 监管条文的引用关系是「义务指向」而非「语义相似」。例如「依照第X条」往往指向程序性要求或罚则，与被引用条文的内容关联度可能很低。无条件扩展会以 0.7 高分注入候选池，稀释真正相关的条文。

#### 5.4.4 Rerank 精排

- 使用百炼 `gte-rerank`，输入为 (原始营销文案, chunk_text) 对
- 候选池（RRF 融合结果）送入 rerank，输出精排分数
- 取 Top-5 作为最终送入 LLM 的 context
- **保存每个 chunk 的 rerank 分数**，作为置信度计算输入

#### 5.4.5 检索结果过滤（修正）

- **取消「全部低于阈值直接返回未检索到相关条文」的硬拦截策略**
- 改为：无论 rerank 分数如何，始终将 Top-5（或 Top-3）送入 LLM 审核
- 若全部低于阈值（如 0.3），在最终输出中附加警告：「检索置信度低，结论仅供参考」
- 同时触发 §5.6.3 的「合规时强制压低置信度」逻辑

> **修正原因：** 金融营销违规中存在大量「低频高损」场景（如新型误导话术、监管未明确列举的擦边表述），这些场景的 query-条文语义距离天然较远，rerank 分数容易低于阈值。直接拒绝检索等于系统对这类文案默认「合规」，与「避免没找到条文=一定合规」的设计意图矛盾。

---

### 5.5 置信度计算

> LLM 不参与置信度打分，全部由系统内部可观测量计算。

#### 5.5.1 四个分量定义（修正）

| 分量 | 计算方式 |
|------|----------|
| `rerank_score` | Top-5 rerank 分数的加权组合：0.7 × mean + 0.3 × max，归一化到 0-1 |
| `chunk_coverage` | Top-5 中 rerank 分数 > 0.3 的 chunk 数量 / 5 |
| `multi_query_agreement` | 多个改写 query 检索结果集合的 Jaccard 相似度 |
| `semantic_diversity` | Top-5 chunk 向量两两平均距离的倒数（衡量检索结果是否聚焦） |

> **修正说明：**
> - `rerank_score` 原设计只取 Top-1，若 Top-1 异常高分但其余不相关，仍会输出高置信度。改为 Top-5 分布特征更鲁棒。
> - `chunk_coverage` 原设计依赖违规类型标签的准确性，存在错误传播风险。改为基于 rerank 分数阈值，脱离标签依赖。
> - `multi_query_agreement` 原设计「多个 query 均检索到同一条文」在短文本场景中区分度不足。改为 Jaccard 相似度更合理。
> - **删除 `violation_type_match`**：原设计为"LLM 输出是否在枚举表中命中"，但 LLM 输出已被枚举表约束，该信号几乎恒为 1，存在循环论证，不贡献区分度。替换为 `semantic_diversity`，衡量检索结果是否聚焦在相关主题上。

#### 5.5.2 加权合成

```
confidence = w1 * rerank_score
           + w2 * chunk_coverage
           + w3 * multi_query_agreement
           + w4 * semantic_diversity
```

- 初始权重：w1=0.4, w2=0.2, w3=0.2, w4=0.2
- **权重在评估阶段用 golden set 校准**（详见 §5.7）

#### 5.5.3 否定词检测（新增）

- 在置信度计算前增加一层语义方向校验：
  - 检测文案中是否含否定词（"不保本""非保证收益""无风险承诺"等）
  - 若检索到禁止性条文但文案含否定表述，则 confidence 强制乘以衰减系数 0.5
  - 该衰减与 §5.6.3 的「合规压低」逻辑叠加

> **新增原因：** 这是原设计最大的盲区——"高置信度但实际上错了"的典型场景。例如文案说"本产品不保本"，检索到"禁止承诺保本"的条文，rerank 分数高、chunk coverage 高，但语义方向完全相反。

#### 5.5.4 合规结论为「合规」时的特殊处理

- 若 rerank_score 全部低于阈值导致 context 质量差，合规结论置信度强制压低（≤ 0.4）
- 在输出中标注：「因未检索到高置信度相关条文而判定合规，结论可信度低」
- 避免「没找到条文 = 一定合规」的误判

### 5.6 LLM 审核

#### 5.6.1 System Prompt 设计

- 角色定义：金融合规审核专家
- 附入 §3.1.1 违规类型枚举表（完整）
- 明确任务：仅根据提供的条文 context 判断，禁止引用 context 之外的条文
- 输出格式强约束（JSON schema）：

```json
{
  "compliant": "yes/no",
  "violations": [
    {
      "violation_type_id": "V02",
      "violation_type_name": "承诺确定收益",
      "article_id": "第二十一条",
      "doc_name": "保险销售行为管理办法",
      "article_text": "条文原文",
      "reason": "具体违规说明",
      "severity": "critical",
      "directional_advice": "建议删除收益确定性表述，改为预期收益范围并附加风险提示"
    }
  ]
}
```

> **新增字段说明：**
> - `severity`（critical/warning/info）：critical 对应可能触发行政处罚的行为（如保本承诺、伪造数据），warning 对应一般用语违规，info 对应建议优化项。让业务用户优先处理高风险项。
> - `directional_advice`：每条 violation 增加一句方向性修改建议，不生成完整文案。降低用户使用门槛，形成"发现问题→知道怎么改"的闭环。

#### 5.6.2 User Prompt 组装

- 结构：营销文案 + 分隔符 + Top-5 条文（按 rerank 分数从高到低，每条附编号、来源文档、正文）
- **新增：** 在条文列表后附加「参考条文」区块（图扩展结果，标注为引用上下文）

#### 5.6.3 输出解析与兜底

- JSON 解析失败触发重试（最多 2 次，prompt 中强化 JSON 格式要求）
- 最终仍失败则返回结构化错误信息

#### 5.6.4 置信度附加

- 将 §5.5 计算的 confidence 值附加到最终输出 JSON 中
- LLM 输出结果与置信度合并为最终响应

#### 5.6.5 置信度-行动映射（新增）

| 置信度区间 | 建议行动 |
|-----------|---------|
| ≥ 0.8 | 可采信，按结论执行 |
| 0.5 – 0.8 | 可发布但需留痕，建议复核 |
| < 0.5 | 建议人工复核，结论仅供参考 |

> 在最终输出中附带此映射建议，让系统输出直接指导业务决策。

#### 5.6.6 错误处理与降级策略（新增）

**API 调用超时/失败：**

| 服务 | 超时 | 重试 | 降级 |
|------|------|------|------|
| Embedding | 30s | 指数退避，最多 3 次 | 失败条文记录日志，不阻断整体流程 |
| Rerank | 15s | 指数退避，最多 3 次 | 跳过 rerank，用 RRF 分数直接取 Top-5 |
| LLM 审核 | 30s | 指数退避，最多 3 次 | 返回 fallback JSON（见下） |

**JSON 解析失败兜底（第 3 次仍失败）：**

```json
{
  "compliant": "unknown",
  "violations": [],
  "error": "llm_output_unparseable",
  "raw_output": "<原始输出文本>",
  "confidence": 0,
  "reason": "LLM 输出格式异常，无法解析"
}
```

**检索为空时的处理：**
- 若检索结果为空（FAISS 返回 0 条），不调用 LLM，直接返回：
```json
{
  "compliant": "unknown",
  "violations": [],
  "confidence": 0.2,
  "reason": "未检索到相关条文"
}
```

### 5.7 评估模块

#### 5.7.1 Golden Set 构建

- **修正：** 明确 30 条为 demo 最小验证集，P1 阶段扩展至 100 条以上
- 人工构造测试样本：
  - 明确违规（≥ 50%）：覆盖枚举表中至少 8 种违规类型，每类 ≥ 3 例（100 条标准）或 ≥ 1 例（30 条 demo）
  - 明确合规（≥ 30%）：措辞规范、含法定风险提示
  - 边界模糊（≥ 20%）：考察置信度是否合理降低
- 每条标注：`expected_compliant`、`expected_article_ids`、`expected_violation_type_ids`

> **修正原因：** 30 条样本的 95% 置信区间约为 [0.61, 0.92]，即使达到 24/30 的正确率，也无法 statistically 证明系统达标。100 条以上才能支撑"系统可用"的判断。

#### 5.7.2 评估指标（修正）

| 指标 | 含义 | 目标值 |
|------|------|--------|
| 合规判断准确率 | yes/no 与标注一致的比例 | — |
| **违规 Recall** | 实际违规被检出的比例 | > 0.90（硬性门槛） |
| **违规 Precision** | 检出违规中实际违规的比例 | — |
| 条文命中率 | 引用条文 ID 与标注的交集 / 标注总数 | — |
| 违规类型准确率 | 输出的 violation_type_id 是否与标注一致 | — |
| 置信度校准度 | 高置信度 bin 的准确率是否单调递增 | Reliability Diagram |

> **修正说明：**
> - 拆分「合规判断准确率」为 `recall_violation` 和 `precision_violation`。金融合规场景中漏报（实际违规但判合规）的代价远高于误报，必须单独监控。
> - 置信度校准改用 Reliability Diagram：将样本按置信度分 5 个 bin，每个 bin 计算实际准确率，绘制 reliability diagram，要求高置信度 bin 的准确率单调递增。

#### 5.7.3 评估流程

- 批量跑完 golden set，结果存为 JSON
- 自动计算六项指标并输出报告
- **新增：** 输出违规类型的混淆矩阵，特别关注「判为合规但实际是 V01」和「V01 判为 V02」两类错误
- 输出错误案例列表，便于定位是 chunking / 检索 / prompt 哪个环节的问题

#### 5.7.4 迭代闭环

- 条文命中率低 → 优化 chunking 粒度或 Query 改写 prompt
- 违规类型准确率低 → 检查枚举表定义是否清晰，优化 LLM system prompt
- 置信度失真 → 用 golden set 重新校准 w1-w4 权重
- 合规判断准确率低 → 增加 few-shot 或调整 rerank 阈值
- **新增：** 否定词误判率高 → 优化否定检测规则或 LLM prompt 中的否定词识别指令

## 6. 已知局限与风险

| 风险 | 说明 | 缓解措施 |
|------|------|---------|
| 枚举表覆盖盲区 | 新型违规手法可能落入 V00 | 定期更新枚举表，V00 样本积累至一定量后人工归纳新类型 |
| 监管文档更新 | 条文修订需重建索引和更新枚举表 | 离线阶段设计为可重复执行，支持增量更新 |
| 隐含语义依赖 | 知识图谱仅覆盖显式引用 | Demo 阶段接受此局限，P2 阶段引入语义关联 |
| 边界模糊案例 | 合规/违规边界模糊时系统应输出低置信度 | 通过 §5.5.3 否定检测和 §5.5.4 合规压低降低误判 |
| 长尾违规漏检 | 低频新型话术可能检索不到 | §5.4.5 取消硬拦截，始终送 LLM 审核并标注低置信度 |
| 样本量不足 | 30 条 demo 样本无法可靠校准权重 | 明确 demo 阶段权重为启发值，P1 扩展至 100+ 条后校准 |

## 8. 审阅反馈汇总

本设计经以下三个视角交叉审阅：

| 视角 | 评分 | 核心结论 |
|------|------|---------|
| 业务合规 | 3.5/5 → 4.5/5（修正后） | 枚举表覆盖度和置信度语义校验为最大短板，修正后业务闭环完整 |
| 检索架构 | 3/5 → 4/5（修正后） | Chunk 边界和阈值策略为 CRITICAL 缺陷，修正后检索链路合理 |
| 工程实现 | 评估中 | 技术选型合理，双路索引在 demo 阶段可落地 |

**关键修正项（按影响排序）：**
1. **Chunk 边界：条 → 条+款** — 避免无关信息淹没相关款
2. **取消阈值硬拦截** — 避免长尾违规被系统默认"合规"
3. **枚举表扩展 V07-V11** — 覆盖隐瞒费用、诱导退保等高频红线
4. **置信度删除 violation_type_match，增加否定检测** — 消除循环论证，解决语义方向误判
5. **图扩展改为条件性参考** — 避免义务指向引用污染精排结果
6. **RRF k=60 → 20** — 适配小语料库场景
7. **增加 severity + directional_advice** — 让输出直接指导业务行动
8. **Golden Set 明确分阶段（30→100）** — 避免用 demo 数据过度承诺
