# 合规审核 RAG 系统 — 高层架构设计

> 本文档面向技术团队与业务干系人，聚焦系统"是什么"和"为什么"，不涉及具体实现细节。

---

## 1. 系统定位

### 1.1 背景与目标

金融营销文案须符合《保险销售行为管理办法》《金融产品网络营销管理办法》《互联网保险业务监管办法》等监管要求。人工逐条审核效率低、标准不一致。

**本系统目标：** 给定一段营销文案，自动定位潜在违规内容、引用对应监管条文、给出合规结论，并附上系统自评的置信度分数。

### 1.2 系统边界

```
输入：营销文案（文本）
         │
         ▼
  ┌─────────────┐
  │ 合规审核RAG  │
  └─────────────┘
         │
         ▼
输出：合规/违规结论 + 引用条文 + 违规类型 + 置信度
```

**系统不做：** 自动修改文案、对接审批流程、实时监控流量。

---

## 2. 核心架构

系统由五个主要模块组成，形成完整的审核闭环：

```
┌─────────────────────────────────────────────────────────┐
│                      离线阶段                            │
│                                                         │
│  监管文档  ──▶  [文档解析与切割]  ──▶  [知识索引构建]    │
│  违规枚举表 ─────────────────────────────────────────▶  │
└─────────────────────────────────────────────────────────┘
                                          │
                                          ▼
┌─────────────────────────────────────────────────────────┐
│                      在线阶段                            │
│                                                         │
│  营销文案 ──▶ [Query理解] ──▶ [混合检索+精排] ──▶ [LLM审核] ──▶ 结论
│                                      │                  │
│                                      └──▶ [置信度计算] ──┘
└─────────────────────────────────────────────────────────┘
```

### 模块职责概述

| 模块 | 职责 | 产出 |
|------|------|------|
| 文档解析与切割 | 将监管文档解析为条文级别的知识单元 | 结构化条文库 |
| 知识索引构建 | 建立向量索引、关键词索引、条文引用图谱 | 可检索知识库 |
| Query 理解 | 将营销文案转化为面向监管的检索意图 | 结构化检索请求 |
| 混合检索+精排 | 召回相关条文并按相关性排序 | Top-N 条文候选 |
| LLM 审核 | 基于检索到的条文给出合规判断 | 结构化审核结论 |
| 置信度计算 | 从系统内部可观测信号合成可信度分数 | 置信度分值 |

---

## 3. 关键设计决策

### 3.1 封闭违规类型枚举表

**决策：** 人工预定义 10–20 个违规类型（V01–V20），系统全链路共享这张表。

**为什么：**
- 让检索、审核、评估三个阶段使用统一语言，避免语义漂移
- 约束 LLM 输出范围，减少幻觉
- 为评估提供精确的 ground truth 对照

**权衡：** 枚举表需人工维护，监管政策更新时需同步修订。

---

### 3.2 混合检索（语义 + 关键词）

**决策：** 同时使用向量检索（语义相似）和 BM25 检索（关键词匹配），结果通过 RRF 融合。

**为什么：**
- 监管文本含大量专业术语，关键词检索捕捉精确措辞
- 违规行为表述多样，向量检索覆盖语义近义表达
- 单路检索在合规场景中召回率不足

**权衡：** 维护两套索引，系统复杂度略高。

---

### 3.3 条文引用知识图谱

**决策：** 解析条文间的显式引用关系（"依照第X条"），构建有向图，检索时做一跳图扩展。

**为什么：**
- 监管条文存在大量交叉引用，仅检索表面命中条文会丢失关联义务
- 图扩展能自动补充「被引用」的上下文条文

**权衡：** 仅处理显式引用，隐含语义关联暂不覆盖。

---

### 3.4 系统自计算置信度（不依赖 LLM 自评）

**决策：** 置信度由系统内部四个可观测信号加权合成，LLM 不参与打分。

**为什么：**
- LLM 自我评估置信度不可靠（容易过度自信）
- 系统信号（检索分数、多 Query 一致性、条文覆盖度）客观且可解释
- 为业务干系人提供可信的"可信度仪表盘"

**权衡：** 初始权重需用标注数据校准。

## 4. 数据流

```
营销文案
  │
  ▼
[Query 理解]
  将文案意图映射为违规类型 + 检索短语
  │
  ▼
[混合检索]
  语义检索 ──┐
              ├──▶ RRF 融合 ──▶ 图扩展 ──▶ [精排]
  关键词检索─┘
  │
  ▼
Top-N 相关条文
  │
  ├──▶ [LLM 审核] ──▶ 违规类型 + 引用条文 + 合规结论
  │
  └──▶ [置信度计算] ──▶ 置信度分值
  │
  ▼
最终输出：结论 + 引用条文 + 违规类型 + 置信度
```

## 5. 质量保障

### 5.1 评估框架

系统配套一个评估模块，基于人工标注的 Golden Set（约 30 条样本）衡量四项指标：

| 指标 | 含义 |
|------|------|
| 合规判断准确率 | yes/no 与人工标注一致的比例 |
| 条文命中率 | 引用条文与标注的重合度 |
| 违规类型准确率 | 输出类型与标注一致的比例 |
| 置信度校准度 | 高置信度样本的准确率是否显著高于低置信度样本 |

### 5.2 迭代方向

评估结果直接指向可优化的模块：

- 条文命中率低 → 优化切割粒度或 Query 改写
- 违规类型偏差 → 优化枚举表定义或 LLM prompt
- 置信度失准 → 重新校准信号权重
- 合规判断偏差 → 调整检索阈值或增加样本

## 6. 建设优先级

```
P0 — 核心审核链路（必须跑通）
  违规枚举表 → 条文解析与切割 → 双路索引
  → Query 改写 → 混合检索 → 精排 → LLM 审核 + 置信度输出

P1 — 质量提升（显著提升结论可信度）
  条文引用解析 → 知识图谱 → 图扩展检索 → 评估模块

P2 — 迭代优化（演示加分项）
  置信度特殊处理（无条文时强制压低） → 检索阈值过滤 → 评估迭代闭环
```

## 7. 已知局限与风险

| 风险 | 说明 |
|------|------|
| 枚举表覆盖盲区 | 新型违规手法可能落入 V00，无法精确分类 |
| 监管文档更新 | 条文修订需重建索引和更新枚举表 |
| 隐含语义依赖 | 知识图谱仅覆盖显式引用，隐含关联暂不处理 |
| 边界模糊案例 | 合规/违规边界模糊时，系统应输出低置信度而非强行判断 |
